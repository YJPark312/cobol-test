## 1. 개요

본 문서는 기업집단 현황 업데이트 시스템(WP-018)에 대한 포괄적인 테스트 케이스를 정의합니다. 시스템은 기존 기업집단 관계를 처리하고 배치 처리를 통해 기업집단 내 모든 관계기업의 포괄적인 금융 및 신용 정보를 업데이트합니다.

### 테스트 범위
- THKIPA110 및 THKIPA111 테이블의 기업집단 관계 배치 처리
- 포괄적인 정보 수집을 위한 8개 업무 모듈과의 통합
- 커밋 간격 관리를 통한 데이터베이스 업데이트 작업
- 오류 처리 및 복구 시나리오
- 연말 처리 특수 케이스

### 테스트 접근법
- 유효한 배치 처리 시나리오에 대한 긍정 테스트 케이스
- 오류 조건 및 무효 데이터에 대한 부정 테스트 케이스
- 경계 조건 및 한계에 대한 경계 테스트 케이스
- 모듈 간 기능에 대한 통합 테스트 케이스

## 2. 테스트 구성

### 테스트 카테고리
- **긍정 테스트**: 성공적인 결과가 예상되는 유효한 처리 시나리오
- **부정 테스트**: 오류 조건 및 무효 입력 시나리오
- **경계 테스트**: 경계 케이스 및 한계 테스트
- **통합 테스트**: 모듈 간 및 시스템 통합 시나리오

### 테스트 데이터 구성
- 긍정 테스트를 위한 유효한 기업집단 데이터 세트
- 부정 테스트를 위한 무효 및 잘못된 형식의 데이터 세트
- 한계 테스트를 위한 경계값 데이터 세트
- 성능 테스트를 위한 대용량 데이터 세트

## 3. 테스트 우선순위

### 중요 우선순위
- 핵심 배치 처리 기능
- 데이터베이스 무결성 및 트랜잭션 관리
- 필수 업무 모듈 통합

### 높은 우선순위
- 오류 처리 및 복구 메커니즘
- 연말 처리 시나리오
- 주채무계열그룹 분류

### 중간 우선순위
- 성능 최적화 시나리오
- 상세 검증 규칙
- 포괄적인 로깅 및 모니터링

### 낮은 우선순위
- 비즈니스 영향이 최소한인 경계 케이스 시나리오
- 선택적 모듈 통합 실패

## 4. 테스트 의존성

### 전제조건
- 테스트 데이터가 포함된 THKIPA110 및 THKIPA111 테이블
- 업무 모듈(XIAA0019, XIIH0059, XIIIK083, XDINA0V2, XIJL4010, XIIBAY01, XIIEZ187, XIIF9911) 사용 가능
- 데이터베이스 연결 및 트랜잭션 지원
- 배치 실행을 위한 JCL 환경

### 테스트 순서 의존성
- 데이터베이스 설정 테스트는 처리 테스트 전에 완료되어야 함
- 모듈 통합 테스트는 개별 모듈 가용성에 의존
- 오류 처리 테스트는 특정 오류 조건이 설정되어야 함

## 5. 테스트 케이스

### TC-WP018-001: 성공적인 기업집단 배치 처리
- **유형:** 긍정
- **우선순위:** 중요
- **설명:** 유효한 데이터로 기업집단 관계의 성공적인 배치 처리 검증
- **기능:** [F-018-001, F-018-008, F-018-009]
- **엔티티:** [BE-018-001, BE-018-002]
- **전제조건:**
  - THKIPA110에 유효한 기업집단 관계 레코드 포함
  - THKIPA111에 유효한 기업집단 정의 레코드 포함
  - 모든 업무 모듈이 사용 가능하고 기능적
- **테스트 데이터:**
  - **입력:**
    - 그룹회사코드: 'KB0'
    - 작업기준일자: '20241219'
    - 심사고객식별자: '1234567890'
    - 기업집단그룹코드: 'G01'
  - **예상 출력:**
    - 포괄적인 금융 정보로 업데이트된 THKIPA110 레코드
    - 집계된 그룹 총계로 업데이트된 THKIPA111 레코드
    - 성공적인 완료를 보여주는 처리 통계
- **테스트 단계:**
  1. 유효한 매개변수로 배치 처리 환경 초기화
  2. 기업집단 관계를 위한 CUR_A110 커서 열기
  3. 모든 업무 모듈을 통해 각 관계기업 레코드 처리
  4. 통합된 정보로 THKIPA110 업데이트
  5. CUR_A111 커서를 통해 기업집단 집계 처리
  6. 그룹 총계 및 분류로 THKIPA111 업데이트
  7. 트랜잭션 커밋 및 완료 통계 생성
- **검증 포인트:**
  - 모든 THKIPA110 레코드가 현재 금융 데이터로 업데이트됨
  - 모든 THKIPA111 레코드가 올바른 집계 총계로 업데이트됨
  - SQL 오류 또는 트랜잭션 실패 없음
  - 처리 통계가 예상 개수와 일치
- **의존성:** 없음

### TC-WP018-002: SOHO 고객 분류 처리
- **유형:** 긍정
- **우선순위:** 높음
- **설명:** 소매 분류를 가진 SOHO(Small Office Home Office) 고객의 올바른 처리 검증
- **기능:** [F-018-002, F-018-008]
- **엔티티:** [BE-018-001]
- **전제조건:**
  - 고유번호구분이 ('01','03','04','05','10','16')인 고객
  - XIIH0059 모듈이 SOHO 익스포저 분류 = '1' 반환
- **테스트 데이터:**
  - **입력:**
    - 고객고유번호구분: '01'
    - 심사고객식별자: '1234567890'
  - **예상 출력:**
    - 기업규모구분: '2' (중소기업)
    - 신용등급구분: SOHO 모듈에서 가져옴
    - 산업분류코드: SOHO 모듈에서 가져옴
- **테스트 단계:**
  1. 개인 고객 분류로 고객 정보 검색
  2. SOHO 분류를 위해 XIIH0059 모듈 호출
  3. SOHO 익스포저 분류 = '1' (소매) 확인
  4. 기업규모구분을 '2' (중소기업)로 설정
  5. 신용등급 및 산업분류 추출
  6. SOHO 특정 정보로 THKIPA110 업데이트
- **검증 포인트:**
  - SOHO 고객이 올바르게 식별되고 분류됨
  - 기업규모가 중소기업 분류로 설정됨
  - SOHO 모듈에서 신용 정보가 채워짐
- **의존성:** [TC-WP018-001]

### TC-WP018-003: CRS 신용평가 통합
- **유형:** 긍정
- **우선순위:** 높음
- **설명:** 기업 신용평가를 위한 CRS(Credit Rating System)와의 통합 검증
- **기능:** [F-018-003, F-018-008]
- **엔티티:** [BE-018-001]
- **전제조건:**
  - 기업 고객 (SOHO 소매 아님)
  - XIIIK083 모듈 사용 가능하고 기능적
- **테스트 데이터:**
  - **입력:**
    - 처리구분코드: '01'
    - 심사고객식별자: '1234567890'
  - **예상 출력:**
    - 최종신용등급구분: CRS에서 가져옴
    - 표준산업분류코드: CRS에서 가져옴
    - 기업규모구분: CRS 규모에서 변환됨
- **테스트 단계:**
  1. 고객이 CRS 처리가 필요한지 결정 (SOHO 소매 아님)
  2. 처리구분 '01'로 XIIIK083 모듈 호출
  3. 신용등급, 산업분류, 규모분류 검색
  4. CRS 규모분류를 내부 기업규모로 변환
  5. CRS 정보로 THKIPA110 업데이트
- **검증 포인트:**
  - CRS 모듈이 성공적으로 호출되고 유효한 데이터 반환
  - 신용등급 및 산업분류가 적절히 추출됨
  - 규모분류가 내부 형식으로 올바르게 변환됨
- **의존성:** [TC-WP018-001]

### TC-WP018-004: 총익스포저 정보 통합
- **유형:** 긍정
- **우선순위:** 중요
- **설명:** 포괄적인 신용 정보를 위한 TE(Total Exposure) 시스템과의 통합 검증
- **기능:** [F-018-005, F-018-008]
- **엔티티:** [BE-018-001]
- **전제조건:**
  - XIJL4010 모듈 사용 가능하고 기능적
  - 고객이 TE 시스템에 신용 한도 보유
- **테스트 데이터:**
  - **입력:**
    - 심사고객식별자: '1234567890'
  - **예상 출력:**
    - 총여신금액: 모든 신용 한도의 합계
    - 여신잔액: 모든 신용 잔액의 합계
    - 고객관리부점코드: TE 시스템에서 가져옴
    - 모든 신용 유형에 대한 한도별 한도 및 잔액
- **테스트 단계:**
  1. 총익스포저 정보를 위해 XIJL4010 모듈 호출
  2. 포괄적인 신용 한도 및 잔액 추출
  3. 고객관리부점 정보 검색
  4. 모든 한도 유형에 걸쳐 총익스포저 계산
  5. 상세한 신용 정보로 THKIPA110 업데이트
- **검증 포인트:**
  - 모든 신용 한도 유형이 적절히 캡처됨
  - 총액이 올바르게 계산되고 업데이트됨
  - 고객관리부점이 올바르게 할당됨
- **의존성:** [TC-WP018-001]

### TC-WP018-005: 연말 처리 시나리오
- **유형:** 긍정
- **우선순위:** 높음
- **설명:** 전년도 총여신금액에 대한 특별 연말 처리 검증
- **기능:** [F-018-008]
- **엔티티:** [BE-018-001]
- **전제조건:**
  - 처리 연도가 이전 처리 연도와 다름
  - 현재 연도 데이터가 있는 기존 THKIPA110 레코드
- **테스트 데이터:**
  - **입력:**
    - 현재 처리 연도: '2024'
    - 이전 처리 연도: '2023'
    - 현재 총여신금액: 1000000000
  - **예상 출력:**
    - 전년총여신금액: 현재 연도 금액으로 업데이트됨
- **테스트 단계:**
  1. 처리에서 연도 변경 감지
  2. 기존 THKIPA110 레코드 검색
  3. 현재 총여신금액을 전년도 필드로 이동
  4. 연말 처리 플래그로 레코드 업데이트
- **검증 포인트:**
  - 전년도 금액이 올바르게 업데이트됨
  - 연말 처리 로직이 적절히 실행됨
  - 현재 연도 데이터가 보존됨
- **의존성:** [TC-WP018-001]
### TC-WP018-006: 주채무계열그룹 분류
- **유형:** 긍정
- **우선순위:** 높음
- **설명:** GRS 등록 기업집단에 대한 주채무계열그룹 분류 검증
- **기능:** [F-018-009]
- **엔티티:** [BE-018-002]
- **전제조건:**
  - 등록코드 'GRS'인 기업집단
  - 기업군관리그룹구분 '01'
- **테스트 데이터:**
  - **입력:**
    - 기업집단등록코드: 'GRS'
    - 기업군관리그룹구분: '01'
  - **예상 출력:**
    - 주채무계열그룹여부: '1' (예)
- **테스트 단계:**
  1. GRS 등록된 기업집단 처리
  2. 기업군관리그룹구분 평가
  3. 규제 기준에 따른 주채무계열그룹 상태 결정
  4. 주채무계열그룹 분류로 THKIPA111 업데이트
- **검증 포인트:**
  - 주채무계열그룹 상태가 올바르게 결정됨
  - 규제 준수 요구사항 충족
  - THKIPA111이 적절한 분류로 업데이트됨
- **의존성:** [TC-WP018-001]

### TC-WP018-007: 빈 고객식별자 처리
- **유형:** 부정
- **우선순위:** 높음
- **설명:** 배치 처리에서 빈 고객식별자 처리 검증
- **기능:** [F-018-001]
- **엔티티:** [BE-018-001]
- **전제조건:**
  - 빈 심사고객식별자가 있는 THKIPA110 레코드
- **테스트 데이터:**
  - **입력:**
    - 심사고객식별자: SPACE
  - **예상 출력:**
    - 처리에서 레코드 건너뜀
    - 건너뜀 카운터 증가
    - 이 레코드에 대한 데이터베이스 업데이트 없음
- **테스트 단계:**
  1. 빈 고객식별자가 있는 레코드 발견
  2. 고객 정보 검증 규칙 적용
  3. 처리 건너뛰고 건너뜀 카운터 증가
  4. 다음 레코드로 계속
- **검증 포인트:**
  - 빈 고객식별자가 적절히 감지됨
  - 레코드가 처리 없이 올바르게 건너뜀
  - 건너뜀 카운터가 정확히 유지됨
- **의존성:** [TC-WP018-001]

### TC-WP018-008: 업무 모듈 호출 실패
- **유형:** 부정
- **우선순위:** 중요
- **설명:** 업무 모듈 호출 실패의 우아한 처리 검증
- **기능:** [F-018-002, F-018-003, F-018-004, F-018-005, F-018-006, F-018-007]
- **엔티티:** [BE-018-001]
- **전제조건:**
  - 하나 이상의 업무 모듈이 사용 불가능하거나 오류 반환
- **테스트 데이터:**
  - **입력:**
    - 심사고객식별자: '1234567890'
