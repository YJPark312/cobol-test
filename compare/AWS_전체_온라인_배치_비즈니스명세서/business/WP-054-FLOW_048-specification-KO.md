## 1. 개요

본 워크패키지는 기업집단심사보고서조회 시스템을 구현하며, 기업집단 신용평가 보고서에 대한 포괄적인 조회 기능을 제공합니다. 시스템은 다년도 역사적 분석을 위한 개별 및 합산연결 재무제표를 포함한 상세한 재무정보를 검색하고 표시합니다.

시스템은 여러 데이터 소스에 걸친 복잡한 재무 데이터 검색 및 처리를 관리합니다:
- DIPA321에서 기업집단 평가보고서 데이터 검색
- 3년 기간에 걸친 계열사 개별 재무제표 데이터
- 공식 기반 처리를 사용한 합산연결 재무제표 계산
- 재무비율 계산 및 단위 변환
- 여러 데이터베이스 테이블 및 계산 엔진과의 통합

주요 업무 목적은 신용평가 담당자가 다음을 수행할 수 있도록 하는 것입니다:
- 포괄적인 기업집단 평가보고서 검색 (기업집단심사보고서 조회)
- 계열사 개별 재무제표 분석 (계열사 개별재무제표 분석)
- 계산된 비율을 포함한 합산연결 재무제표 검토 (합산연결재무제표 및 비율 검토)
- 다년도 재무동향 분석 지원 (다년도 재무동향 분석)
- 단위 변환 기능을 갖춘 표준화된 재무보고 제공 (표준화된 재무보고 및 단위변환)

시스템은 현재 평가년도와 이전 2년의 데이터를 처리하여 기업집단 신용평가 결정을 위한 포괄적인 역사적 재무분석 기능을 제공합니다.

  - 평가기준년월일은 유효한 YYYYMMDD 형식이며 미래 일자가 아니어야 함
  - 단위는 재무금액 스케일링 팩터를 결정하며 '3' 또는 '4'여야 함
  - 처리사용자ID는 시스템의 유효한 활성 사용자여야 함
  - 처리단말기ID는 인증된 단말기여야 함
  - 요청타임스탬프는 현재 시스템 타임스탬프 형식이어야 함

| 오류심각도수준 | String | 1 | 1-5 | 오류 심각도 수준 (1=치명적, 5=정보) | WK-ERR-SEV-LVL | errSevLvl |
| 오류메시지 | String | 200 | NOT NULL | 상세 오류메시지 | WK-ERR-MSG | errMsg |
| 오류컨텍스트 | String | 500 | 선택사항 | 오류 컨텍스트 정보 | WK-ERR-CNTXT | errCntxt |
| 오류타임스탬프 | String | 20 | YYYYMMDDHHMMSSNNNNNN | 오류 발생 타임스탬프 | WK-ERR-TMSTMP | errTmstmp |
| 오류발생모듈 | String | 8 | NOT NULL | 오류가 발생한 모듈 | WK-ERR-SRC-MOD | errSrcMod |
| 오류발생라인 | Numeric | 6 | 양수 | 오류가 발생한 라인 번호 | WK-ERR-SRC-LINE | errSrcLine |
| 사용자ID | String | 8 | NOT NULL | 오류와 연관된 사용자 | WK-ERR-USER-ID | errUserId |
| 단말기ID | String | 8 | NOT NULL | 오류와 연관된 단말기 | WK-ERR-TERM-ID | errTermId |
| 거래ID | String | 12 | NOT NULL | 오류와 연관된 거래 | WK-ERR-TXN-ID | errTxnId |
| 복구조치 | String | 100 | 선택사항 | 취해진 복구 조치 | WK-RECOV-ACT | recovAct |
| 해결상태 | String | 2 | NOT NULL | 오류 해결 상태 | WK-RESOL-STAT | resolStat |
| 해결일자 | String | 8 | YYYYMMDD | 오류가 해결된 일자 | WK-RESOL-DT | resolDt |
| 에스컬레이션수준 | String | 1 | 0-3 | 오류 에스컬레이션 수준 | WK-ESC-LVL | escLvl |

- **검증 규칙:**
  - 오류인스턴스ID는 각 오류 발생에 대해 고유해야 함
  - 오류코드는 유효한 시스템 오류코드여야 함
  - 오류범주는 유효한 분류(DB, BZ, SY 등)여야 함
  - 오류심각도수준은 1과 5 사이여야 함
  - 오류타임스탬프는 정밀한 타임스탬프 형식이어야 함
  - 오류발생모듈은 유효한 시스템 모듈이어야 함
  - 사용자ID와 단말기ID는 유효한 시스템 식별자여야 함
  - 해결상태가 해결됨을 나타낼 때 해결일자가 필요함

## 3. 업무 규칙

### BR-054-001: 입력검증규칙 (Input Validation Rule)
- **설명:** 포괄적인 필드 검증을 통한 처리 전 필수 입력 매개변수 검증
- **조건:** 입력 매개변수가 수신되면 모든 필수 필드 및 업무 제약조건을 검증
- **관련 엔티티:** [BE-054-001]
- **예외사항:** 
  - 기업집단그룹코드가 비어있으면 처리 UKIP0001과 함께 오류 B3600552 반환
  - 기업집단등록코드가 비어있으면 처리 UKIP0002와 함께 오류 B3600552 반환  
  - 평가년월일이 비어있으면 처리 UKIP0003과 함께 오류 B2700019 반환
  - 처리사용자ID가 무효하면 처리 UKIP0004와 함께 오류 B3800004 반환
  - 처리단말기ID가 인증되지 않으면 처리 UKIP0005와 함께 오류 B3800005 반환

### BR-054-002: 다년도처리규칙 (Multi-Year Processing Rule)
- **설명:** 데이터 일관성 검증을 통한 현재년도 및 이전 2년의 재무데이터 처리
- **조건:** 평가기준일자가 제공되면 기준년도로 끝나는 연속 3년을 계산하고 데이터 가용성을 검증
- **관련 엔티티:** [BE-054-003, BE-054-004]
- **예외사항:** 
  - 년도 계산이 음수 또는 무효한 년도를 초래하지 않아야 함
  - 어떤 년도에 대해 재무데이터를 사용할 수 없으면 부분 처리와 함께 경고 B4200001 반환
  - 년도 범위가 시스템 한계를 초과하면 처리 UKIP0006과 함께 오류 B2700020 반환

### BR-054-003: 단위변환규칙 (Unit Conversion Rule)
- **설명:** 정밀도 검증을 통한 지정된 표시단위에 따른 재무금액 변환
- **조건:** 표시단위가 '3'이면 금액을 1,000으로 나누고, 표시단위가 '4'이면 금액을 100,000,000으로 나눔
- **관련 엔티티:** [BE-054-003, BE-054-004]
- **예외사항:** 
  - 계산 오류를 피하기 위해 금액이 0이 아닐 때만 나누기 적용
  - 변환이 정밀도 손실을 초래하면 반올림 규칙 적용
  - 변환된 금액이 필드 한계를 초과하면 처리 UKIP0007과 함께 오류 B3000108 반환

### BR-054-004: 데이터원천매핑규칙 (Data Source Mapping Rule)
- **설명:** 검증을 통한 재무분석 데이터원천 코드를 설명적 이름으로 매핑
- **조건:** 재무분석 데이터원천 코드가 수신되면 적절한 원천명으로 매핑하고 원천 가용성을 검증
- **관련 엔티티:** [BE-054-003]
- **예외사항:** 
  - 코드 = '1'이면 원천 = '신용평가' (Credit Evaluation)
  - 코드 = '2'이면 원천 = '한신평(개별FS)' (Hansin Individual FS)
  - 코드 = '3'이면 원천 = '크렙탑(개별FS)' (Crebtop Individual FS)
  - 코드가 인식되지 않으면 원천 = '-'이고 경고 로그
  - 데이터원천을 사용할 수 없으면 처리 UKIP0008과 함께 오류 B4200002 반환

### BR-054-005: 개별재무제표처리규칙 (Individual Financial Statement Processing Rule)
- **설명:** 데이터 무결성 검증을 통한 다년도에 걸친 각 계열사의 개별 재무제표 처리
- **조건:** 계열사가 식별되면 검증과 함께 3년간 각 회사의 개별 재무제표를 검색
- **관련 엔티티:** [BE-054-003]
- **예외사항:** 
  - 재무분석결산구분코드는 '1'로 고정
  - 처리구분은 개별재무제표를 위해 '02'로 고정
  - 일자 형식은 연말을 위해 '1231'을 월일로 하는 YYYYMMDD여야 함
  - 재무제표 데이터가 불완전하면 데이터 품질 규칙 적용
  - 회사가 비활성이면 알림과 함께 처리 건너뜀

### BR-054-006: 합산연결재무제표계산규칙 (Consolidated Financial Statement Calculation Rule)
- **설명:** 포괄적인 검증을 통한 공식 기반 처리를 사용한 합산연결 재무제표 계산
- **조건:** 합산연결 재무데이터가 필요하면 사전 정의된 계산 코드 및 검증과 함께 공식 파싱 엔진 사용
- **관련 엔티티:** [BE-054-004, BE-054-005]
- **예외사항:**
  - 기준년도와 결산년도가 적절히 계산되어야 함
  - 재무분석결산구분코드 = '1'
  - 재무분석보고서구분1 = '11', 구분2 = '17'
  - 모델계산대분류 = 'YQ'
  - 모델계산소분류 코드: '0001'부터 '0012'까지
  - 모델적용일자 = '20191224'
  - 계산구분 = '11'
  - 공식 처리가 실패하면 처리 UKIP0009와 함께 오류 B3000825 반환
  - 연결 엔티티가 일치하지 않으면 처리 UKIP0010과 함께 오류 B4200003 반환

### BR-054-007: 산식처리규칙 (Formula Processing Rule)
- **설명:** 포괄적인 오류 처리를 통한 FIPQ001 공식 파싱 엔진을 사용한 재무 공식 처리
- **조건:** 공식 내용이 수신되면 검증과 함께 처리구분 '99'를 사용하여 파싱 및 계산 결과
- **관련 엔티티:** [BE-054-004, BE-054-005]
- **예외사항:** 
  - 공식 파싱 오류는 우아하게 처리되어야 함
  - 결과는 적절한 숫자 형식으로 변환되어야 함
  - 개별 공식 계산이 실패해도 처리 계속
  - 공식 구문이 무효하면 처리 UKIP0011과 함께 오류 B3000826 반환
  - 계산 오버플로우가 발생하면 처리 UKIP0012와 함께 오류 B3000827 반환
  - 0으로 나누기가 감지되면 기본값 적용하고 경고 로그

### BR-054-008: 재무제표유형분류규칙 (Financial Statement Type Classification Rule)
- **설명:** 검증을 통한 재무제표를 개별 또는 합산연결로 분류
- **조건:** 개별 회사 데이터 처리 시 유형을 '개별'로 설정, 그룹 수준 데이터 처리 시 유형을 '합산연결'로 설정
- **관련 엔티티:** [BE-054-003, BE-054-004]
- **예외사항:** 
  - 유형 분류는 데이터 원천 및 처리 방법과 일치해야 함
  - 분류가 데이터와 충돌하면 처리 UKIP0013과 함께 오류 B4200004 반환

### BR-054-009: 그리드출력관리규칙 (Grid Output Management Rule)
- **설명:** 용량 검증을 통한 재무제표 표시를 위한 출력 그리드 행 수 및 인덱싱 관리
- **조건:** 재무제표 데이터가 처리되면 용량 확인과 함께 그리드 행 수를 증가시키고 적절한 인덱싱 관리
- **관련 엔티티:** [BE-054-003, BE-054-004, BE-054-006]
- **예외사항:** 
  - 그리드 인덱싱은 모든 재무제표 항목에 걸쳐 순차적이고 일관되어야 함
  - 그리드 용량이 초과되면 처리 UKIP0014와 함께 오류 B3000109 반환
  - 인덱싱 불일치가 감지되면 그리드 재설정 및 재구축

### BR-054-010: 오류처리규칙 (Error Handling Rule)
- **설명:** 포괄적인 로깅 및 복구를 통한 처리 중 다양한 오류 조건 처리
- **조건:** 시스템 오류가 발생하면 로깅과 함께 적절한 오류 코드 및 처리 메시지 반환
- **관련 엔티티:** [모든 엔티티, BE-054-008]
- **예외사항:**
  - 데이터베이스 접근 오류는 처리 UKII0182와 함께 B3900002 반환
  - 프로그램 호출 실패는 호출된 프로그램의 특정 오류 코드 반환
  - 공식 파싱 오류는 FIPQ001 반환 코드로 처리
  - 치명적 오류가 발생하면 복구 절차 시작
  - 오류 임계값이 초과되면 시스템 관리자에게 에스컬레이션

### BR-054-011: 데이터품질검증규칙 (Data Quality Validation Rule)
- **설명:** 포괄적인 품질 지표를 통한 재무정보의 데이터 품질 표준 보장
- **조건:** 재무데이터가 처리되면 데이터 품질 지표 및 완전성 표준을 검증
- **관련 엔티티:** [BE-054-003, BE-054-004, BE-054-007]
- **예외사항:**
  - 데이터 완전성이 임계값 미만이면 경고 B4200005 반환
  - 데이터 정확성 문제가 감지되면 데이터 정제 규칙 적용
  - 데이터 일관성 위반이 발견되면 처리 UKIP0015와 함께 오류 B4200006 반환

### BR-054-012: 보안감사규칙 (Security and Audit Rule)
- **설명:** 모든 운영에 대한 보안 준수 및 포괄적인 감사추적 보장
- **조건:** 어떤 운영이 수행되면 보안 권한을 검증하고 감사추적을 기록
- **관련 엔티티:** [BE-054-001, BE-054-008]
- **예외사항:**
  - 사용자가 필요한 권한이 없으면 처리 UKIP0016과 함께 오류 B3800006 반환
  - 감사추적 기록이 실패하면 처리 UKIP0017과 함께 오류 B3900009 반환
  - 보안 위반이 감지되면 에스컬레이션 및 계정 잠금

### BR-054-013: 성능모니터링규칙 (Performance Monitoring Rule)
- **설명:** 임계값 관리를 통한 시스템 성능 및 처리 효율성 모니터링
- **조건:** 처리 성능이 저하되면 성능 최적화 절차 트리거
- **관련 엔티티:** [BE-054-006]
- **예외사항:**
  - 처리 시간이 임계값을 초과하면 경고 B3000110 반환
  - 메모리 사용량이 한계를 초과하면 처리 최적화
  - CPU 사용률이 치명적이면 비중요 작업 연기

### BR-054-014: 통화변환규칙 (Currency Conversion Rule)
- **설명:** 환율 검증을 통한 다중 통화 재무데이터의 통화 변환 처리
- **조건:** 통화 변환이 필요하면 검증과 함께 현재 환율 적용
- **관련 엔티티:** [BE-054-003]
- **예외사항:**
  - 환율을 사용할 수 없으면 처리 UKIP0018과 함께 오류 B3001447 반환
  - 통화코드가 무효하면 처리 UKIP0019와 함께 오류 B3001448 반환
  - 변환이 상당한 차이를 초래하면 승인 필요

### BR-054-015: 영업일자검증규칙 (Business Date Validation Rule)
- **설명:** 영업일자를 검증하고 휴일/주말 처리를 처리
- **조건:** 일자 필드가 처리되면 영업 달력에 대해 검증
- **관련 엔티티:** [BE-054-001, BE-054-002, BE-054-003]
- **예외사항:**
  - 일자가 휴일이면 이전 영업일 사용
  - 일자가 주말이면 이전 영업일 사용
  - 일자가 무효하면 처리 UKIP0020과 함께 오류 B2700021 반환

### BR-054-016: 계산정밀도규칙 (Calculation Precision Rule)
- **설명:** 모든 재무계산에 걸친 계산 정밀도 및 반올림 일관성 보장
- **조건:** 재무계산이 수행되면 일관된 정밀도 및 반올림 규칙 적용
- **관련 엔티티:** [BE-054-003, BE-054-004, BE-054-005]
- **예외사항:**
  - 정밀도 손실이 감지되면 은행가 반올림 적용
  - 계산 오버플로우가 발생하면 처리 UKIP0021과 함께 오류 B3000828 반환
  - 언더플로우가 발생하면 경고와 함께 0으로 설정

### BR-054-017: 데이터보존규칙 (Data Retention Rule)
- **설명:** 재무데이터의 데이터 보존 기간 및 아카이브 요구사항 관리
- **조건:** 데이터 보존 기간이 만료되면 규제 요구사항에 따라 데이터 아카이브
- **관련 엔티티:** [BE-054-002, BE-054-003, BE-054-004]
- **예외사항:**
  - 아카이브가 실패하면 오류 로깅과 함께 재시도
  - 보존 기간이 충돌하면 가장 긴 필요 기간 사용
  - 규제 요구사항이 변경되면 보존 정책 업데이트

### BR-054-018: 통합일관성규칙 (Integration Consistency Rule)
- **설명:** 통합 시스템 및 데이터 원천 간 일관성 보장
- **조건:** 데이터 통합이 수행되면 모든 통합 시스템 간 일관성 검증
- **관련 엔티티:** [BE-054-002, BE-054-003, BE-054-004]
- **예외사항:**
  - 통합 불일치가 감지되면 처리 UKIP0022와 함께 오류 B3002140 반환
  - 시스템 동기화가 실패하면 복구 절차 시작
  - 데이터 버전 불일치가 있으면 최신 일관된 버전 사용

### BR-054-019: 백업복구규칙 (Backup and Recovery Rule)
- **설명:** 재무데이터 처리를 위한 백업 및 복구 절차 정의
- **조건:** 시스템 장애가 발생하면 데이터 무결성 검증과 함께 복구 절차 실행
- **관련 엔티티:** [모든 업무 엔티티]
- **예외사항:**
  - 백업을 사용할 수 없으면 처리 UKIP0023과 함께 오류 B3900010 반환
  - 복구가 실패하면 치명적 시스템 오류로 에스컬레이션
  - 데이터 무결성이 손상되면 수동 검증 필요

### BR-054-020: 규제준수규칙 (Regulatory Compliance Rule)
- **설명:** 재무보고 규제 및 표준 준수 보장
- **조건:** 재무보고서가 생성되면 규제 요구사항 준수 검증
- **관련 엔티티:** [BE-054-002, BE-054-003, BE-054-004]
- **예외사항:**
  - 준수 위반이 감지되면 처리 UKIP0024와 함께 오류 B4200007 반환
  - 규제 표준이 변경되면 검증 규칙 업데이트
  - 준수 감사가 실패하면 경영진 승인 필요

  - 인덱스: 그룹 코드 및 계산 분류의 기본키, 성능을 위한 공식 내용 인덱스

### 6.5 기술 아키텍처 및 통합

#### 6.5.1 시스템 아키텍처 계층
- **AS 계층 (Application Server)**: AIP4A32 - 포괄적인 업무 로직 조율을 통한 기업집단 평가보고서 조회를 위한 주요 애플리케이션 서버 프로그램
- **IC 계층 (Interface Component)**: IJICOMM - 공통 처리 초기화, 보안 검증 및 시스템 자원 관리를 위한 인터페이스 구성요소
- **DC 계층 (Data Component)**: DIPA321, DIPA521 - 포괄적인 데이터 검증을 통한 평가보고서 및 재무제표 검색을 위한 데이터 구성요소
- **SQLIO 계층**: QIPA313, QIPA52D - 성능 최적화를 통한 계열사 및 합산연결 재무데이터 쿼리를 위한 데이터베이스 접근 구성요소
- **프레임워크 계층**: YCCOMMON - 포괄적인 로깅 기능을 통한 오류 처리, 감사추적 관리 및 시스템 통합을 위한 공통 프레임워크

#### 6.5.2 데이터 플로우 아키텍처
1. **입력 플로우**: AIP4A32 → IJICOMM → YCCOMMON → 입력 검증 → 보안 인증
2. **데이터베이스 접근**: AIP4A32 → DIPA321/DIPA521 → QIPA313/QIPA52D → 데이터베이스 테이블 → 데이터 검색
3. **서비스 호출**: AIP4A32 → FIPQ001 → 공식 파싱 엔진 → 수학적 계산 → 검증된 결과
4. **출력 플로우**: 데이터베이스 결과 → 재무계산 → 단위변환 → 데이터 품질 평가 → 출력 그리드 → 응답 생성
5. **오류 처리**: 모든 계층 → YCCOMMON 프레임워크 → 오류 분류 → 복구 절차 → 사용자 응답
6. **감사추적**: 모든 작업 → 보안 프레임워크 → 감사추적 기록 → 준수 보고 → 관리 대시보드
