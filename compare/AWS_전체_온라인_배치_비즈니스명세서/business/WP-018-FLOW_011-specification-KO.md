## 1. 개요

본 워크패키지는 기업집단 현황 업데이트 시스템의 배치 프로세스를 구현하여 현재 기업집단 관계 정보를 유지관리합니다. 시스템은 기존 기업집단 데이터를 처리하고 관련 회사들의 포괄적인 금융 및 신용 정보를 업데이트합니다.

### 비즈니스 목적
시스템은 다음을 통해 기업집단 관계를 유지관리합니다:
- 관계기업기본정보(관계기업기본정보)를 현재 금융 데이터로 업데이트
- 기업관계연결정보(기업관계연결정보)를 집계된 총액으로 갱신
- 여러 업무 모듈에서 신용평가 데이터 통합
- 신용한도, 잔액, 담보를 포함한 포괄적인 금융 지표 유지
- 모든 기업집단 관리 시스템 간 데이터 일관성 보장

### 처리 개요
1. THKIPA110 및 THKIPA111 테이블에서 기존 기업집단 관계 조회
2. 각 관계기업에 대해 여러 업무 모듈에서 포괄적인 정보 수집:
   - 신용등급 및 업종분류를 위한 CRS(신용평가시스템) 정보
   - 개인고객을 위한 SOHO(소상공인) 분류
   - DINA0V2 모듈의 기업여신정책 정보
   - 신용한도 및 잔액을 위한 TE(총익스포져) 정보
   - 대출잔액을 위한 사후관리(사후관리) 정보
   - 담보금액을 위한 통합담보(통합담보) 정보
   - 위험관리를 위한 조기경보시스템 정보
3. THKIPA110을 포괄적인 금융 및 신용 정보로 업데이트
4. THKIPA111을 집계된 그룹 총액 및 주채무계열그룹 분류로 업데이트
5. 성능 최적화를 위해 커밋 간격으로 데이터를 배치 처리

### 주요 비즈니스 개념
- **관계기업(관계기업)**: 통합 신용평가가 필요한 기업집단의 일부인 회사들
- **기업집단그룹코드(기업집단그룹코드)**: 기업집단의 고유 식별자
- **심사고객식별자(심사고객식별자)**: 신용심사 대상 고객의 고유 식별자
- **주채무계열그룹(주채무계열그룹)**: 특별한 규제 처리가 필요한 주요 채무관계 그룹
- **총익스포져(총익스포져)**: 모든 신용시설 및 파생상품을 포함한 포괄적인 신용노출
- **여신정책(여신정책)**: 특정 고객 또는 그룹에 적용되는 기업 대출정책 및 제한사항

  - 주채무그룹 결정은 대기업집단에 대한 규제 가이드라인을 따름

## 3. 비즈니스 규칙

### BR-018-001: 고객정보 검증 규칙
- **설명:** 포괄적인 업데이트 처리 전에 고객이 유효한 식별정보와 사업자등록을 가지고 있는지 검증
- **조건:** 심사고객식별자 > SPACE일 때 포괄적인 정보 수집을 진행
- **관련 엔티티:** [BE-018-001]
- **예외사항:** 
  - 고객 식별자가 비어있으면 처리를 건너뛰고 건너뛰기 카운터 증가
  - 고객고유번호를 가져올 수 없으면 제한된 처리로 계속
- **비즈니스 로직:** 유효한 심사고객식별자를 가진 고객만 포괄적인 정보 업데이트 대상

### BR-018-002: 고객유형 분류 규칙
- **설명:** 고객고유번호 분류에 따라 처리 방식을 결정 (개인 vs 법인)
- **조건:** 고객고유번호구분이 ('01','03','04','05','10','16') 중 하나일 때 개인고객으로 처리, 그렇지 않으면 법인고객으로 처리
- **관련 엔티티:** [BE-018-001]
- **예외사항:** 
  - 개인고객(SOHO)은 다른 신용평가 처리를 받음
  - 법인고객은 전체 CRS(신용평가시스템) 처리를 받음
- **비즈니스 로직:** 고객 유형에 따라 정보 수집을 위해 호출되는 업무 모듈이 결정됨

### BR-018-003: 신용규모 분류 변환 규칙
- **설명:** CRS 신용평가 규모 분류를 내부 기업규모 분류로 변환
- **조건:** CRS 신용평가규모분류가 제공될 때 사전 정의된 매핑을 사용하여 내부 규모 분류로 변환
- **관련 엔티티:** [BE-018-001]
- **예외사항:** 
  - 알 수 없는 규모 분류는 기본값 '0'(없음)으로 설정
  - SOHO 고객은 자동으로 '2'(중소기업)로 분류
- **비즈니스 로직:** CRS 규모 분류는 일관된 보고를 위해 내부 기업규모 분류로 매핑됨

### BR-018-004: 연말처리 규칙
- **설명:** 연말 운영 시 전년총여신금액에 대한 특별 처리
- **조건:** 처리 연도가 이전 처리 연도와 다를 때 전년총여신금액을 업데이트
- **관련 엔티티:** [BE-018-001]
- **예외사항:** 
  - 동일 연도 처리 시 기존 전년 금액 유지
  - 최초 처리 시 전년 금액을 0으로 초기화
- **비즈니스 로직:** 전년총여신금액은 전년 대비 분석 및 규제 보고를 위해 보존됨

### BR-018-005: 주채무계열그룹 분류 규칙
- **설명:** 기업군관리그룹구분에 따라 주채무그룹 상태를 결정
- **조건:** 기업군관리그룹구분 = '01' AND 등록코드 = 'GRS'일 때 주채무그룹 상태를 평가
- **관련 엔티티:** [BE-018-002]
- **예외사항:** 
  - 수기등록 그룹('수기')은 자동으로 주채무그룹으로 분류되지 않음
  - 주채무그룹 상태는 규제 자본 요구사항에 영향을 미침
- **비즈니스 로직:** 주채무그룹은 특별한 규제 처리 및 모니터링 요구사항을 받음

### BR-018-006: 포괄적 정보 통합 규칙
- **설명:** 여러 업무 모듈의 정보를 통합하여 포괄적인 고객 프로필 생성
- **조건:** 고객 처리가 시작될 때 모든 관련 업무 모듈을 순차적으로 호출
- **관련 엔티티:** [BE-018-001]
- **예외사항:** 
  - 모듈 실패는 로그에 기록되지만 처리를 중단하지 않음
  - 누락된 정보는 기본값으로 우아하게 처리됨
- **비즈니스 로직:** 포괄적인 고객 정보는 CRS, DINA0V2, TE, 사후관리, 담보, 조기경보 시스템의 통합이 필요함

### 기술 아키텍처

- **BATCH 계층**: BIP0002 - 기업집단 현황 업데이트 처리를 위한 JCL 작업 제어가 포함된 메인 배치 처리 프로그램으로, 포괄적인 업무 모듈 정보 수집을 통합
- **SQLIO 계층**: RIPA110, RIPA111 - THKIPA110(관계기업기본정보) 및 THKIPA111(기업관계연결정보) 테이블 작업을 위한 DBIO 인터페이스 서비스를 제공하는 데이터베이스 액세스 컴포넌트
- **프레임워크**: YCCOMMON, YCDBIOCA, YCDBSQLA - 트랜잭션 제어, 데이터베이스 작업, SQL 인터페이스 서비스, 포괄적인 시스템 통합을 위한 에러 처리를 제공하는 공통 프레임워크 컴포넌트
- **모듈 통합**: CRS, SOHO, TE, 사후관리, 담보, 조기경보 시스템에 걸친 포괄적인 고객 정보 수집을 위한 8개 전문 모듈(XIAA0019, XIIH0059, XIIIK083, XDINA0V2, XIJL4010, XIIBAY01, XIIEZ187, XIIF9911)을 지원하는 업무 모듈 통합 계층

### 데이터 플로우 아키텍처

1. **입력 플로우**: JCL SYSIN → BIP0002 → 매개변수 검증(그룹회사코드, 작업기준일자) → 처리 환경 초기화
2. **데이터베이스 액세스**: BIP0002 → RIPA110/RIPA111 → THKIPA110/THKIPA111 테이블 → 커밋 간격 관리를 통한 커서 기반 순차 처리
3. **모듈 통합**: BIP0002 → 순차적 업무 모듈 호출 → 다중 시스템에서 정보 수집:
   - 고객정보: XIAA0019 → 고객고유번호 및 분류
   - SOHO 분류: XIIH0059 → 소매고객 분류 및 신용등급
   - CRS 정보: XIIIK083 → 기업신용평가 및 업종분류
   - 여신정책: XDINA0V2 → 기업여신정책 및 제한사항
   - 총익스포져: XIJL4010 → 포괄적인 신용한도 및 잔액
   - 사후관리: XIIBAY01 → 대출잔액 및 연체정보
   - 담보: XIIEZ187 → 통합담보 회복가치
   - 조기경보: XIIF9911 → 위험관리 정보
4. **출력 플로우**: BIP0002 → 데이터베이스 업데이트(THKIPA110/THKIPA111) → 처리 통계 생성 → 로그 파일 출력 → JCL 완료
5. **에러 처리**: 모든 계층 → 프레임워크 에러 처리 → SQLCODE 모니터링 → 우아한 에러 복구 → 에러 코드 반환(12-99)
