## 1. 개요

본 워크패키지는 관계기업 기본정보를 지주사 전송용으로 변환하고 암호화하는 배치 처리 시스템을 구현합니다. 시스템은 관계기업 및 기업집단의 고객 데이터를 처리하고, 데이터 보안을 위한 암호화를 적용하며, 검증 체크섬이 포함된 전송 파일을 생성합니다.

업무 목적:
- 관계기업기본정보 고객정보암호화 변환 처리
- A110(고객데이터) 및 A111(기업집단데이터) 암호화 전송파일 생성
- 규제 준수를 위한 그룹사고객정보제공통지 처리
- 일방향 및 양방향 암호화 메커니즘을 통한 데이터 보안 보장

## 3. 업무 규칙

### BR-002-001: 그룹회사 필터링 규칙
- **설명:** KB금융그룹 레코드만 처리
- **조건:** 고객 또는 기업집단 데이터 처리 시 그룹회사코드 = 'KB0'로 필터링
- **관련 엔티티:** [BE-002-001, BE-002-002]
- **예외:** 다른 그룹회사코드를 가진 레코드는 처리에서 제외

### BR-002-002: 고객유형 제외 규칙
- **설명:** 개인고객을 처리에서 제외
- **조건:** 고객고유번호구분 = '01'인 경우 처리에서 제외
- **관련 엔티티:** [BE-002-001]
- **예외:** ('07','08','09','13','15') 유형의 법인고객만 처리

### BR-002-003: 고객데이터 마스킹 규칙
- **설명:** 개인정보 보호를 위한 개인고객 데이터 마스킹
- **조건:** 고객고유번호구분 = '01'인 경우 대표고객명과 대표사업자등록번호를 SPACE로 설정
- **관련 엔티티:** [BE-002-001]
- **예외:** 법인고객은 실제 명칭과 사업자번호 유지

### BR-002-004: 쉼표 치환 규칙
- **설명:** CSV 형식 문제 방지를 위한 고객명 내 쉼표 치환
- **조건:** 고객명 처리 시 모든 쉼표를 공백으로 치환
- **관련 엔티티:** [BE-002-001, BE-002-002]
- **예외:** 예외 없음 - 모든 고객명에 적용

### BR-002-005: 고객고유번호 암호화 규칙
- **설명:** 고객고유번호에 솔트를 포함한 일방향 암호화 적용
- **조건:** 고객고유번호 처리 시 THKIIK923에서 조회한 서비스 ID를 사용하여 솔트 포함 일방향 암호화 적용
- **관련 엔티티:** [BE-002-001, BE-002-003]
- **예외:** 암호화 성공 필수, 실패 시 처리 중단

### BR-002-006: 레코드 암호화 규칙
- **설명:** 완전한 데이터 레코드에 양방향 암호화 적용
- **조건:** 출력 파일 생성 시 양방향 암호화 서비스를 사용하여 완전한 레코드 암호화
- **관련 엔티티:** [BE-002-001, BE-002-002, BE-002-003]
- **예외:** 암호화 성공 필수, 실패 시 레코드 미작성

### BR-002-007: 법인등록번호 할당 규칙
- **설명:** 고객유형에 따른 법인등록번호 할당
- **조건:** 고객고유번호구분 ≠ '01'인 경우 고객고유번호를 법인등록번호에 할당, 그렇지 않으면 SPACE 할당
- **관련 엔티티:** [BE-002-001]
- **예외:** 개인고객(유형 '01')은 SPACE 값 할당

### BR-002-008: 배치 커밋 규칙
- **설명:** 성능을 위한 1000건마다 데이터베이스 트랜잭션 커밋
- **조건:** 레코드 수가 1000으로 나누어떨어질 때 트랜잭션 커밋
- **관련 엔티티:** [BE-002-001, BE-002-002]
- **예외:** 처리 종료 시 건수와 관계없이 최종 커밋 수행

### BR-002-009: 고객통지 규칙
- **설명:** 규제 준수를 위한 고객정보제공통지 삽입
- **조건:** 각 고객 레코드 처리 시 THKJLG001에 통지 레코드 삽입
- **관련 엔티티:** [BE-002-001]
- **예외:** 중복 통지(SQLCODE +803/-803)는 건너뜀

### BR-002-010: 통지일자 계산 규칙
- **설명:** 기준처리일자에 따른 통지일자 계산
- **조건:** 기준일자 < '20160301'이면 '20150529' 사용, 기준일자가 '20160301'과 '20161231' 사이면 '20160301' 사용, 그 외에는 기준년도 + '0101' 사용
- **관련 엔티티:** [BE-002-001]
- **예외:** 예외 없음 - 일자 계산 필수

### BR-002-011: 검증ID 생성 규칙
- **설명:** 데이터 무결성을 위한 순차적 검증 ID 생성
- **조건:** 레코드 처리 시 1부터 시작하는 순차적 검증 ID 할당
- **관련 엔티티:** [BE-002-001, BE-002-002]
- **예외:** 각 레코드 유형(A110/A111)은 별도 순서 유지

### BR-002-012: 파일 형식 규칙
- **설명:** 특정 필드 순서의 쉼표 구분 값 형식 생성

- **관련 엔티티:** [BE-002-001, BE-002-002]
- **예외:** 예외 없음 - 형식 고정

## 4. 업무 기능

### F-002-001: 처리환경 초기화
- **설명:** 시스템 매개변수 초기화 및 입력 요구사항 검증
- **입력:**

| 입력 | 데이터 타입 | 길이 | 제약조건 | 설명 |
|------|-------------|------|----------|------|
| 그룹회사코드 | String | 3 | Fixed='KB0' | 시스템 그룹회사 식별자 |
| 작업수행년월일 | Date | 8 | YYYYMMDD 형식 | 처리 기준일자 |
| 배치작업구분 | String | 3 | IN ('ZAD','ZAB','ZAP') | 배치 시스템 구분 |

- **출력:**

| 출력 | 데이터 타입 | 길이 | 제약조건 | 설명 |
|------|-------------|------|----------|------|
| 초기화상태 | String | 2 | '00'=성공 | 초기화 결과 |
| 일방향서비스ID | String | 12 | NOT NULL | 암호화 서비스 식별자 |
| 양방향서비스ID | String | 12 | NOT NULL | 암호화 서비스 식별자 |

- **처리 로직:**
  - JCL SYSIN에서 시스템 입력 매개변수 수락
  - 작업기준일자가 공백이 아닌지 검증
  - 출력 파일 오픈 (A110.DAT, A110.CHK, A111.DAT, A111.CHK)
  - 시스템 구분에 따라 THKIIK923에서 암호화 서비스 ID 조회
  - 카운터 및 작업 변수 초기화
- **적용 업무규칙:** [BR-002-001]

### F-002-002: 관계기업정보 처리
- **설명:** 고객데이터 암호화를 포함한 관계기업 기본정보 추출 및 변환
- **입력:**

| 입력 | 데이터 타입 | 길이 | 제약조건 | 설명 |
|------|-------------|------|----------|------|
| 관계기업데이터 | Record | Variable | THKIPA110 + THKAABPCB에서 | 고객 및 금융 데이터 |

- **출력:**

| 출력 | 데이터 타입 | 길이 | 제약조건 | 설명 |
|------|-------------|------|----------|------|
| 암호화A110레코드 | String | 428 | 암호화 형식 | 암호화된 고객데이터 레코드 |
| A110검증레코드 | String | 28 | CSV 형식 | 검증 체크섬 레코드 |

- **처리 로직:**
  - 커서 CUR_C001 실행하여 관계기업 데이터 조회
  - 그룹회사코드 'KB0' 및 고객유형 ('07','08','09','13','15')으로 필터링
  - 개인고객(유형 '01')에 대한 고객데이터 마스킹 적용
  - 고객명의 쉼표를 공백으로 치환
  - 고객고유번호를 ASCII 형식으로 변환
  - 고객고유번호에 솔트 포함 일방향 암호화 적용
  - 고객유형에 따른 법인등록번호 할당
  - 쉼표 구분자로 완전한 레코드 형식화
  - 완전한 레코드를 ASCII 형식으로 변환
  - 완전한 레코드에 양방향 암호화 적용

### 기술 아키텍처

- **BATCH Layer**: BIP0004 - JCL 작업 제어를 포함한 메인 배치 처리 프로그램
- **SQLIO Layer**: 커서 처리를 통한 DB2 테이블 직접 SQL 액세스
- **Framework**: 공통 처리 및 오류 처리를 위한 YCCOMMON 프레임워크
- **Utilities**: XFAVSCPN (암호화), XZUGCDCV (코드 변환)

### 데이터 흐름 아키텍처

1. **입력 흐름**: JCL SYSIN → BIP0004 → 매개변수 검증
2. **데이터베이스 액세스**: BIP0004 → DB2 커서 → THKIPA110/THKIPA111/THKAABPCB
3. **서비스 호출**: BIP0004 → XFAVSCPN (암호화) → XZUGCDCV (코드 변환)
4. **출력 흐름**: BIP0004 → 암호화 파일 (A110.DAT, A110.CHK, A111.DAT, A111.CHK)
5. **오류 처리**: 모든 계층 → YCCOMMON 프레임워크 → 오류 메시지 및 로깅
